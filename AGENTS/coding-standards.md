# Coding Standards

## 1. 务实原则 (TypeScript 容忍度)

由于服务端 API 数据结构可能存在动态变化，涉及以下数据层时：

- 服务端返回的核心数据
- message 事件 payload
- 第三方 API JSON

**允许**：

- 适度使用 `any` 兜底
- 广泛使用可选链 (`?.`)
- 使用宽松的类型保护

**优先保证**：

- 页面稳定不报错
- 不阻断核心主干渲染
- 不出现不可恢复的运行时崩溃

不要在这里过度追求 TypeScript 类型的“绝对完美”与“深度推导”，业务不挂掉是第一优先级。

## 2. 异步超时防御

涉及状态锁（如 `isLoading`）和长效 Promise 的操作，**必须使用 `setTimeout` 设定合理的超时兜底释放机制**，防止由于网络异常或消息丢失导致的 UI 死锁。

## 3. 异步状态与异常闭环规范

- 所有的 `async/await` 异步操作必须被完整的 `try/catch/finally` 包裹。
- 凡是涉及到 UI 状态锁（如 `isLoading.value = true`）的操作，**必须且只能在 `finally` 块中进行重置**，以保证无论是成功还是请求异常熔断，页面都能恢复可交互状态。

## 4. 函数声明规范

- **组件顶部函数**（如 `<script setup>` 最外层）默认使用 `function` 声明。
- **函数内部**才使用箭头函数（闭包或嵌套函数）。
- **其他的钩子函数**（如生命周期 `onMounted`、计算属性 `computed` 等）使用时默认使用箭头函数。

## 5. 模块导入规范

- **路径别名**：本项目已配置基于 `@/` 映射到 `src` 目录的路径别名。在跨层级或引入通用模块时，请一律使用别名路径进行快速导入（例如：`@/components/Xxx`、`@/utils/Xxx`），严禁使用复杂的多层相对路径（如 `../../../components`）。对于同目录下的相邻文件，允许使用相对路径（如 `./types`）。

## 6. 类型注释规范

- **类型备注的完整性**：定义 TypeScript 类型（如 `interface` 或 `type`）时，**必须为每个字段提供完整的 JSDoc 或单行注释备注**。确保后续在业务代码中使用或悬停在该类型时，开发者能够直观理解字段的具体含义、枚举值的表现以及边界情况。
