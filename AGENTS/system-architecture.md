# System Architecture & Component Design

## 1. 组件分层与封装规范

当前项目采用明确的**组件分层与二次封装**架构，所有的组件开发与修改必须遵循此层次结构，严禁跨层级混乱调用或破坏现有的封装逻辑：

- **基础 UI 组件层 (`src/components/ui/`)**
  仅负责最基础的、无业务状态的原子 UI 组件（例如：基础的 Input、Button、Select）。
  此层组件应保持高度通用性，不耦合具体的业务逻辑。

- **业务/功能组件层 (例如：`src/components/form/` 等)**
  负责处理特定的功能场景。这一层的组件（如表单类的复合组件）**必须基于 `ui/` 层的原子组件进行二次封装**。
  例如：`form` 目录下的组件用于处理表单的状态、校验、布局等逻辑，其内部的输入元素应直接引用 `ui` 层的基础组件，而不是原生的 HTML 元素或重新造轮子。

- **其他特定领域组件 (如 `dialog`, `confirm`, `common` 等)**
  同理，其他维度的组件也应遵循“高层封装低层”的原则，将公共逻辑进行收敛，复用基础的 UI 组件。

**核心原则**：
1. **职责单一**：各级组件各司其职，基础层不管业务，业务层不写基础样式。
2. **单向依赖**：高层业务组件依赖低层 UI 组件，低层 UI 组件禁止反向依赖高层组件。
3. **拒绝重复造轮子**：在编写新组件时，必须优先查看 `ui/` 层是否已有现成可用的基础组件。
