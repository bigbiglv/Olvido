# UI, Theme & i18n Standards

## 1. 主题与样式开发规范

当前项目拥有一套完整的主题系统，日常开发中编写样式时，必须严格遵守以下主题与样式的约束：

- **禁止硬编码颜色与字体**
  严禁在 CSS/Less/SCSS 或行内样式中硬编码十六进制颜色值（如 `#FF0000`）、绝对像素大小等。
  所有的颜色、字体大小、间距等必须使用项目中已定义好的**主题变量（CSS Variables）**或**设计系统（Design System/Tailwind 等）预设的 Token**。
  
- **语义化类名优先**
  在使用样式时，应当使用具有语义的 Token。例如，表示成功状态的颜色应使用类似于 `var(--color-success)` 或 `text-success`，而不是使用具体的色值。这样在切换暗黑模式或更换主题时，才能保证整个应用的 UI 表现一致。

- **组件样式隔离**
  非全局通用样式必须通过 `scoped` 属性进行隔离，防止不同组件之间的样式互相污染。

## 2. 国际化 (i18n) 规范

为了支持多语言并保证前端展示的一致性，所有的 UI 文本必须接入项目的国际化系统：

- **禁止硬编码文案**
  在任何业务组件（Vue 模板、TS 逻辑代码、甚至抛出的错误提示）中，严禁直接书写中文字符串或英文硬编码文案。
  示例（禁止）：`<div>提交成功</div>` 或者 `message.success('提交成功')`。

- **使用多语言 Key**
  所有的文案必须提取到 `src/i18n/` 对应的语言包文件中，并在组件内通过翻译函数（如 `t('form.submitSuccess')`）进行动态渲染。
  示例（推荐）：`<div>{{ t('form.submitSuccess') }}</div>`。

- **动态变量插值**
  当文案中存在动态变量时，严禁使用字符串拼接的方式（因为不同语言的语序可能不同），必须使用 i18n 系统提供的变量插值功能（例如 `t('user.welcome', { name: userName })`）。

**总结**：在输出涉及 UI 的代码时，必须时刻保持“文案走 i18n，样式走 Theme Token”的底线原则。

## 3. Tailwind CSS 规范

本项目目前使用最新的 Tailwind CSS 语法体系（默认视同为 v4 及以上版本规范）。在生成类名或操作样式时，必须严格遵守以下规范：

- **拒绝旧版/废弃语法**
  严禁使用 Tailwind 早期版本中已废弃的类名或不支持的写法。在生成代码前，务必基于最新的 Tailwind 规范输出，坚决杜绝混入 v2/v3 时期的历史旧写法。
  
- **拥抱现代 CSS 变量支持**
  充分利用现代 Tailwind 对 CSS 变量的无缝集成支持。使用自定义颜色或尺寸时，必须优先利用项目中配置好的系统 Theme Tokens。
  尽量减少对“任意值”的使用（如 `w-[123px]`、`text-[#123456]`），除非遇到极特殊的边界场景，否则一律通过系统的语义化 class 或 token 代替。

- **现代特性与简化写法**
  严格遵循最新的响应式修饰符、颜色透明度简写（如 `bg-blue-500/50`）等现代特性。保持类名简洁，避免无脑堆砌臃肿或效果重叠的原子类。
